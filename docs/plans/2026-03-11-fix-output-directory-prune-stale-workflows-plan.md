# fix: prune stale workflows from output directory

## Overview

Ensure `cinzel github parse` keeps the output directory in sync with current HCL sources by removing generated workflow files that no longer exist in source.

## Problem Statement / Motivation

When a workflow is removed or renamed in `cinzel/*.hcl`, regeneration writes current files but leaves stale YAML files in the output directory. This creates drift and can leave obsolete workflows active in `.github/workflows`.

Current behavior is overwrite-only:

- parse builds current in-memory outputs from source (`provider/github/parse_workflow.go`)
- writer writes current targets (`provider/github/github.go`)
- no stale reconciliation step exists

## Proposed Solution

Add stale-output pruning for generated workflow files using provider-aware file markers.

Recommended shape:

1. Add a top marker comment to generated YAML files, for example:
   - `# generated-by: cinzel`
   - `# cinzel-provider: github`
2. On successful parse/write, compute stale candidates as existing `*.yaml` in output directory minus current-generated-set.
3. Delete only stale candidates whose marker proves they were generated by cinzel for the current provider.
4. Skip unmarked files and files marked for other providers.

This avoids deleting non-cinzel files in mixed directories while guaranteeing removed/renamed workflows are cleaned.

## Technical Considerations

- Keep workflow source-of-truth in `cinzel/*.hcl`; never hand-edit generated YAML.
- Pruning must run only after full parse/write success.
- Dry-run mode must not delete files.
- Limit first implementation scope to workflow YAML files (not action subdirectories) to reduce risk.
- Marker parsing must be strict and safe (no deletion on malformed/ambiguous marker state).

## System-Wide Impact

- **Interaction graph**: `internal/command` parse options -> provider parse orchestration (`provider/github/github.go`) -> fs write/prune operations.
- **Error propagation**: prune failures should return actionable errors and avoid partial cleanup.
- **State lifecycle risks**: malformed marker parsing can cause false positives; deletion must fail safe.
- **API surface parity**: CLI, `.cinzelrc` behavior (if supported), and docs must describe identical sync semantics.
- **Integration test scenarios**: remove/rename workflows, mixed-output-dir safety, and dry-run no-op behavior.

## Acceptance Criteria

- [x] Renaming a workflow in HCL creates the new file and removes the old file in the same run.
- [x] Non-cinzel files in output directory are never removed by prune.
- [x] Files generated by other providers are never removed by prune.
- [x] Dry-run mode performs no deletion.
- [x] Re-running parse with unchanged source results in no create/delete changes.
- [x] Workflow regeneration and prune behavior is covered by tests.

## Success Metrics

- No stale generated workflow files remain after regeneration.
- Fewer manual cleanup commits in `.github/workflows` after workflow renames/removals.

## Dependencies & Risks

- **Dependency:** clear ownership model for generated files.
- **Risk:** accidental deletion in mixed directories.
  - **Mitigation:** marker-based ownership; delete only files with cinzel marker for current provider.
- **Risk:** pruning on partial generation failure.
  - **Mitigation:** prune only after full successful parse/write.
- **Risk:** marker drift or manual edits to generated header.
  - **Mitigation:** fail safe by skipping deletion unless markers match exactly.
